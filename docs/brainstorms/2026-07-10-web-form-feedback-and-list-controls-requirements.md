# Web 表单反馈与列表控件体验整改

## Problem Frame

当前 Web 页面中的普通表单在业务校验失败时会直接导航到 JSON 错误响应，用户既丢失当前操作上下文，也无法方便地修改后重试。工作项流转弹窗把当前状态和下一状态混在同一组选项中，语义不清。工作项列表及筛选区同时存在大面积纯白、层级薄弱、筛选项排列松散和原生下拉控件风格不统一的问题。

## Requirements

**表单反馈**
- R1. 普通 Web 表单遇到业务错误时停留在当前页面，以自动消失的消息提示展示服务端错误内容。
- R2. 提交期间应阻止重复操作并给出明确的忙碌状态；成功后继续遵循服务端重定向。
- R3. 登录失效仍跳转登录页；文件直传、确认操作和已有 htmx 表单保持原有专用流程。

**工作项流转**
- R4. “推进并指派”只提供当前状态允许到达的下一状态，不把当前状态作为可选项重复展示。
- R5. 下一状态使用清晰的动作化中文文案，当前状态在控件外单独说明。

**列表与控件**
- R6. 工作项列表和筛选区形成清晰的页面层级，减少无边界的大面积纯白，同时保持安静、紧凑的工作台风格。
- R7. 筛选控件按关键词、条件项和操作区重新排版，在桌面和移动端均不拥挤、不溢出。
- R8. 所有原生下拉控件获得统一的容器、箭头、悬停、焦点和禁用态视觉，并保留浏览器原生表单语义与键盘操作。

## Success Criteria

- 制造一个业务校验错误后，页面不跳转到 JSON，错误消息清晰可见并自动消失。
- 工作项流转下拉不再显示当前状态，选项名称与实际状态迁移动作一致。
- Bug、任务、需求共享列表在桌面和移动视口中层级清晰，筛选区对齐稳定。
- 项目内普通 `select` 外观一致，没有未经设计的原生折叠控件。

## Scope Boundaries

- 本轮不实现替代浏览器原生弹出菜单的完整 ARIA listbox，也不为长列表增加搜索能力。
- 本轮不改变工作项状态机规则，不修改 API JSON 错误协议。
- 本轮不改造登录和初始化页面的认证错误展示流程。

## Key Decisions

- 采用原生 `select` 的统一渐进增强，而不是重写自定义下拉菜单，以保留可访问性和移动端选择体验。
- 在前端共享层拦截普通 Web POST 表单并处理 JSON 错误，避免逐个后端接口重复包装错误页面。
- 工作项三类列表继续复用同一模板，在共享结构上统一调整。
