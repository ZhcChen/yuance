# 当前项目上下文需求

## Problem Frame

元策的需求、任务和 Bug 页面目前偏跨项目聚合。对普通项目成员来说，多个项目的工作项混在一起会降低上下文清晰度，也容易在创建或查看工作项时误选项目。

系统需要提供一个轻量的“当前项目”上下文，让用户在日常工作中先选定项目，再围绕该项目查看和创建需求、任务、Bug。

## Requirements

**当前项目选择**
- R1. 顶部区域需要提供“当前项目”选择入口，展示用户可访问的项目。
- R2. 用户选择当前项目后，系统应在后续页面访问中保留该选择。
- R3. 当前项目选择应按用户隔离，不应影响其他用户。
- R4. 用户只能选择自己可访问的项目；系统管理员可以选择所有项目。

**项目上下文过滤**
- R5. 需求、任务、Bug 列表默认受当前项目过滤。
- R6. 创建需求、任务、Bug 时默认使用当前项目。
- R7. 如果用户未选择当前项目，进入需求、任务、Bug 页面时不应展示跨项目混合列表，应提示用户先选择当前项目。
- R8. 当前项目切换后，需求、任务、Bug 页面应自动进入新项目上下文。

**全局能力边界**
- R9. 项目列表、个人中心、系统管理、审计日志、对象存储设置不受当前项目强制过滤。
- R10. 全局搜索仍可跨项目搜索，但结果必须保留项目标识；后续可增加“仅当前项目”选项。
- R11. 工作台可以保留“我的工作项”等个人跨项目入口，但项目工作区内容应优先围绕当前项目展示。

## Success Criteria

- 普通成员进入需求、任务、Bug 页面时，不再看到多个项目混合的工作项。
- 用户可以在顶部明确知道当前正在处理哪个项目。
- 创建工作项时默认落到当前项目，减少误选项目。
- 用户切换当前项目后，后续列表和创建行为随之变化。
- 系统管理和全局搜索等全局页面不被错误限制。

## Scope Boundaries

- 第一版不做多项目同时选中。
- 第一版不做组织空间、产品线、迭代等更复杂层级。
- 第一版不强制全局搜索只能搜索当前项目。
- 第一版不改造项目成员权限模型，只复用已有项目可见性规则。

## Key Decisions

- 当前项目是“用户工作上下文”，不是数据权限模型本身；权限仍由 RBAC 和项目成员关系共同决定。
- 需求、任务、Bug 页面默认必须依赖当前项目，避免跨项目混合。
- 项目列表和系统管理保留全局视角，避免管理员和负责人失去总览能力。
