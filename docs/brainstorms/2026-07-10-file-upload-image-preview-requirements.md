# 文件上传与图片预览体验

## 问题范围

当前附件直传流程能够完成文件上传，但用户在提交前无法确认图片内容，上传中没有可感知的字节进度，Bug 详情中的截图也只能作为普通附件下载。这会让截图型 Bug 的排查效率和上传可靠性感知都偏低。

## 需求

**上传体验**
- R1. 项目附件、工作项附件、评论附件和新建 Bug 的图片分组上传，均应在选择图片后立即展示本地预览、文件名和大小；普通文件继续使用清晰的文件状态展示。
- R2. 直传对象存储期间应显示稳定尺寸的环形百分比进度，并明确区分登记、签名、上传、完成和失败状态；失败后保留所选文件，允许重试。
- R3. 上传控件应保持现有的权限、文件类型与大小限制，以及签名直传和完成回写流程，不把本地文件持久化到应用服务。

**已上传图片查看**
- R4. 已上传的图片附件应在项目、工作项和评论附件区域展示缩略图；非图片附件保留现有文件行与下载能力。
- R5. 图片缩略图应按需获取短期下载地址，加载、失效和失败状态均需可见，且不得将长期存储凭据暴露给浏览器。
- R6. 点击缩略图或预览入口应打开统一图片查看器，支持适屏、放大、缩小、旋转、重置、键盘 Escape 关闭和相邻图片切换。

## 验收标准

- 用户选择截图后，不上传也能看清将要提交的图片。
- 用户能在上传过程中看到真实的百分比变化，并在失败后直接重试。
- Bug 评论中的已上传截图无需下载即可查看，且可在查看器中放大和旋转。
- 普通文档、压缩包和 pending/deleted 附件的既有行为不回归。

## 范围边界

- 不新增文件存储服务、图片转码、缩略图持久化或多文件队列管理。
- 不改变附件 API 契约、权限模型、对象存储配置或文件大小限制。
- 不在首次版本加入标注、裁剪、编辑图片或视频预览。

## 关键决策

- 本地预览使用浏览器临时 object URL：无需上传前落盘，也能即时确认内容。
- 已上传图片使用现有短期下载地址：沿用受权限保护的附件下载能力，避免模板携带长期可访问地址。
- 图片查看器为全局单例：所有附件区域复用同一个弹窗和控制器，避免每条评论重复插入复杂 DOM。
- 进度使用真实上传字节事件：避免以阶段文字或模拟百分比代替实际传输状态。

## 依赖与假设

- 对象存储的浏览器直传 URL 支持 `PUT` 请求并允许浏览器报告上传进度。
- 现有 `/api/v1/.../download-url` 响应可供登录用户按需取得临时图片地址。
