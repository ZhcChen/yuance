# 元策 MVP 需求边界

## Problem Frame

元策是一个面向企业研发团队的轻量项目管理系统。初期参考禅道的核心管理闭环，但不追求完整复刻，重点让团队能够围绕项目、需求、任务、Bug 和成员协作。

系统初期采用单体部署形态：一个 Rust 服务承载页面、接口、静态资源、模板和业务逻辑，降低部署和运维复杂度。

管理能力不再设计为独立后台。系统只有统一的用户 Web 界面和 API：管理员也是普通用户账号，只是拥有系统管理权限。这样用户模型、登录态、权限判断和页面入口都更简单。

## Requirements

**产品范围**
- R1. 系统初期以“项目”为中心组织功能，一个项目下管理需求、任务、Bug、成员和动态。
- R2. 第一版应覆盖登录、项目列表、项目详情、需求管理、任务管理、Bug 管理、我的工作台。
- R3. 系统管理能力内置在用户 Web 界面中，通过权限控制展示和访问，不单独建设 `/admin` 后台。
- R4. 权限体系采用轻量 RBAC：用户、角色、权限点、用户角色绑定、角色权限绑定；初期内置系统管理员和普通成员角色，不做多 portal 账号域。
- R16. 系统管理区需要提供对象存储设置入口，第一版先支持阿里云 OSS，用于后续项目附件、评论附件、Bug 截图等文件上传能力。
- R20. 首次访问系统时，如果还没有完成初始化，应进入一次性管理员初始化流程，由用户自行填写首个系统管理员的用户名、显示名称和密码；初始化成功后自动进入 `/web`，后续不能重复初始化。
- R21. 开发 / 测试环境允许通过显式 local seed 初始化一个固定超级管理员账号，用于本地联调和自动化测试；正式环境必须禁止该逻辑。

**路由边界**
- R5. 根路径 `/` 默认跳转到 `/web`。
- R6. 所有用户界面和系统管理页面统一以 `/web` 开头。
- R7. 项目不提供 `/admin` 独立后台入口。
- R8. API 接口统一以 `/api` 开头。
- R9. htmx 页面片段归属于 `/web` 命名空间；`/api` 保持为程序化 JSON 接口边界。

**部署与运行时**
- R10. 项目采用多模块仓库，但初期只实现一个 `api` 模块。
- R11. `api` 模块使用 Rust 开发，并负责服务端渲染页面、API、静态资源和业务逻辑。
- R12. HTML、CSS、JS 与 htmx 页面资源应随 Rust 二进制一起打包发布。
- R13. 数据库使用 SQLite。
- R14. 项目不引入 Redis。
- R15. 缓存使用进程内内存缓存，但缓存不得成为唯一数据来源；进程重启后系统应能从 SQLite 恢复核心状态。

**开发规范**
- R17. 项目需要同步参考项目中适合元策的开发规范，包括 Git 提交流程、SQL / repository 分层、服务端分页、管理页面密度、说明提示、迁移不可变、seed 边界和敏感配置治理。
- R18. 同步规范时必须适配元策技术栈，不复制参考项目中的 Go、React、Ant Design、PostgreSQL、Redis、worker 多模块、独立后台等不适用约束。
- R19. 对象存储密钥不得明文写入代码、文档、日志或普通页面展示；运行时配置应进入 SQLite，并通过受控系统管理页面维护。

## Success Criteria

- 用户可以从 `/web` 进入工作界面，完成项目、需求、任务、Bug 的基础管理。
- 拥有对应 RBAC 权限的用户可以在 `/web` 内看到系统管理菜单，并完成用户、角色、权限、系统设置、对象存储设置和审计查看。
- 普通成员看不到系统管理菜单，也不能通过直接访问系统管理 URL 绕过权限。
- 空库首次访问 `/web` 或 `/web/login` 时，用户可以在页面上创建首个系统管理员账号；初始化完成后同一入口切换为正常登录，不再开放初始化。
- 开发 / 测试环境可以显式执行 local seed 创建固定超级管理员；生产环境执行同一命令必须失败，不得创建默认账号。
- 系统管理员可以从 `/web/system/storage` 配置阿里云 OSS 对象存储，并为后续文件上传能力提供统一配置来源。
- 外部或内部程序化调用可以通过 `/api` 路由访问明确的接口边界。
- 应用可以作为单个二进制部署运行，除 SQLite 数据文件外不依赖 Redis 等外部状态服务。
- 进程重启不会造成核心业务数据丢失。

## Scope Boundaries

- 初期不做 Redis、分布式缓存、分布式锁或多实例横向扩展。
- 初期不做独立 `/admin` 后台、不做独立后台用户体系、不做多 portal 账号模型。
- 初期做轻量 RBAC，但不做复杂数据权限矩阵、审批式授权、多 portal 账号域或外部 IAM。
- 正式环境不预置默认管理员账号和默认密码，也不把首个管理员密码写入生产 seed 或运行时环境变量。
- 开发 / 测试环境可以保留明确标注的固定 local seed 超级管理员账号，但该 seed 不能进入生产发布流程。
- 初期不做完整禅道功能集，例如测试用例、测试计划、发布管理、甘特图、燃尽图、复杂报表、DevOps 集成和知识库。
- 初期不采用前后端分离 SPA 架构。
- 初期不拆分多个业务服务或微服务。
- 初期对象存储只接入阿里云 OSS，不同时做 MinIO、S3、七牛、腾讯云 COS 等多供应商适配。
- 初期只建设对象存储配置、探测、上传签名和下载签名基础能力，不在 MVP 第一阶段强制完成所有业务附件场景。

## Key Decisions

- 统一用户模型：系统管理员也是用户，不再区分“后台用户”和“用户端用户”，通过角色和权限控制系统管理能力。
- 轻量 RBAC：参考 qfy-sc 的账号 / 角色 / 权限 / 角色授权 / 权限快照设计，适配为元策单 portal、SQLite、统一 `/web` 用户模型；RBAC 管“功能权限”，项目成员关系管“项目数据范围”。
- 统一页面入口：只保留 `/web` 页面入口和 `/api` JSON 接口入口，避免 `/admin` 与 `/web` 两套页面、认证和权限规则分裂。
- 项目中心模型：比“产品 + 项目 + 测试”三中心更适合 MVP，能降低用户心智和开发复杂度。
- SQLite 作为主存储：符合单体部署目标，减少运维依赖。
- 内存缓存替代 Redis：符合单进程运行模型，但必须避免把缓存当作持久状态。
- 单二进制发布：HTML、CSS、JS 和 htmx 资源随 Rust 服务一起发布，避免前后端版本错配。
- 首个系统管理员自助初始化：参考 qfy-sc 的 bootstrap 思路，但在元策中适配为 `/web` 服务端渲染页面；空库不内置默认密码，初始化成功后创建系统管理员角色绑定、写入初始化完成状态并建立登录 session。
- 开发测试固定超管：参考 qfy-sc 的 `seed local-admins` 思路，元策提供仅限 development / test / local 环境的 local seed，默认账号 `yuance_admin`，默认密码 `Yuance@2026Dev!`，用于当前开发测试和自动化验收；生产环境必须拒绝执行。
- 对象存储配置入库：阿里云 OSS 的 endpoint、region、bucket、AccessKey 等运行态配置由 `/web/system/storage` 写入 SQLite，避免长期依赖环境变量。
- 开发规范同步但不照搬：只吸收参考项目中可复用的工程纪律和质量约束，技术栈、模块形态和路由形态以元策为准。

## Dependencies / Assumptions

- 系统初期以单实例运行为前提。
- SQLite 数据文件需要有明确的数据目录、备份策略和迁移策略，具体实现留到技术计划阶段。
- 用户端和系统管理页面都使用服务端渲染加 htmx 交互，不要求独立前端工程。
- 阿里云 OSS 配置第一版作为系统级配置，由系统管理员维护；后续业务文件上传统一复用该配置。
- 首次管理员初始化以 SQLite 中的初始化状态和管理员角色绑定为准；完成后即使后续管理员被禁用或删除，也不应自动重新开放公开初始化入口。
- RBAC 权限点由 core seed 幂等维护；业务代码只校验权限 key，不在 handler 里硬编码角色名。
- local seed 超级管理员只用于开发 / 测试环境；生产环境首次部署仍必须走 `/web` 自助初始化。
