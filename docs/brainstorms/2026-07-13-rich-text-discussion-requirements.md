# 工作项讨论富文本与截图上传体验

## Problem Frame

测试、产品和开发在需求、任务、Bug 详情中最常见的协作材料是截图、录屏和上下文说明。当前评论和回复依赖纯文本加独立附件选择控件，用户需要先把截图保存成本地文件，再选择上传，路径冗长且附件脱离正文上下文。新的讨论输入应让用户直接粘贴或拖拽截图 / 文件，在正文位置完成上传和展示。

## Requirements

**富文本编辑**
- R1. 工作项评论、回复和评论编辑输入框改为富文本编辑器，支持基础文字排版、换行、链接、列表、图片和文件卡片。
- R2. 评论正文展示为安全富文本，历史纯文本评论继续正常显示。
- R3. 编辑评论时应保留已有富文本内容和内联媒体，保存后继续显示原发表时间和编辑时间。

**粘贴与拖拽上传**
- R4. 用户可直接把剪贴板截图、图片或文件粘贴到富文本编辑器，系统立即创建内联占位并开始上传。
- R5. 用户可把本地图片或文件拖拽到富文本编辑器，系统立即创建内联占位并开始上传。
- R6. 粘贴上传和拖拽上传共用同一上传队列、进度展示、失败状态、重试和移除交互。
- R7. 上传中的图片或文件应在编辑器正文位置显示状态；成功后替换为可预览图片、视频预览入口或文件卡片。
- R8. 上传失败时不得丢失用户已输入文字；失败项应可重试或移除。

**讨论流程**
- R9. 发表、回复、发表并指派、回复并指派的行为保持不变，富文本内容与内联媒体都归属本次评论。
- R10. 评论 / 回复区域移除新增附件选择控件，不再出现独立“添加附件”入口。
- R11. 历史评论附件继续保留展示和下载 / 预览能力，避免旧数据不可见。

**安全与兼容**
- R12. 服务端必须对富文本 HTML 做白名单过滤，只允许安全标签、属性和受控媒体引用。
- R13. 富文本中的图片、视频和文件引用必须经过现有权限校验，不允许任意外部脚本、事件属性或不受控协议。
- R14. 通知摘要、列表摘要和搜索摘要应从富文本中提取纯文本，避免直接显示 HTML。

## Success Criteria

- 用户截图后可直接在评论框粘贴并看到图片上传进度，不需要保存到本地。
- 用户可把文件拖进评论框，文件出现在正文位置并显示上传状态。
- 发表后图片在评论正文中直接展示，并可点击进入现有预览弹窗。
- 上传失败时评论输入内容不丢失，失败媒体可重试或删除。
- 历史纯文本评论和历史附件不回归。
- 富文本内容不能注入脚本、事件属性或危险链接。

## Scope Boundaries

- 第一期优先覆盖工作项评论、回复和评论编辑；新建 / 编辑需求、任务、Bug 主描述暂不一起改造。
- 第一期不做复杂协同编辑、@提及、表情反应、图片标注、裁剪或服务端图片转码。
- 第一期不移除历史附件数据结构和旧附件展示，只移除评论新增附件按钮。
- 视频支持以浏览器原生预览能力为准，不做转码和封面抽帧。

## Key Decisions

- 评论富文本先行：先解决最高频的截图粘贴和讨论上下文问题，降低一次性改造所有图文输入的风险。
- 粘贴和拖拽统一：两种输入方式对用户是同一种“把文件放进正文”的行为，应复用状态、上传和错误处理。
- 历史附件保留：旧评论附件属于协作记录，不能因为新增富文本入口而消失。
