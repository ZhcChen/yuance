# 元策 Codex CLI Agent 输出样例

本文档给 `yuance-agent` 的使用者和维护者参考。

它不定义规则，规则仍以这两份文档为准：

```text
skills/yuance-agent/SKILL.md
docs/mcp/ai-agent-playbook.md
```

本文只回答一件事：

```text
元策场景下，Codex CLI 的实际输出长什么样。
```

## 1. 项目分析样例

适用场景：

- 用户让 Codex CLI 分析某个项目当前推进情况
- 已经读取项目概况、重点工作项和部分评论上下文

示例：

```text
结论：
- YCE 当前主要压力集中在图片上传和工作项通知两条线，Bug 收敛速度正常，但“消息已读”和“图片预览一致性”仍有尾项。

关键证据：
- 进行中的高优先级工作项主要集中在 bug 和 task，且处理人集中在前端实现链路。
- 最近评论里多次提到图片预览、消息角标和指派流转体验，说明当前关注点仍是交互完整性而不是底层接口不可用。

当前风险 / 缺口：
- 图片预览相关改动跨上传、富文本、弹窗三个区域，容易出现样式回归。
- 通知链路虽然已具备基础能力，但如果未读、已读、跳转定位三者不一致，用户体验会显得“不可信”。

建议下一步：
- 先针对图片上传 -> 预览 -> 评论渲染做一轮端到端冒烟。
- 再针对“被指派 / 被回复”消息做一轮消息列表与帖子定位回归。
```

## 2. 工作项分析样例

适用场景：

- 用户给出具体工作项编号
- Codex CLI 已读取详情和评论上下文，但还没有执行写入

示例：

```text
结论：
- YCE-BUG-18 当前不是接口阻塞，而是前端交互收尾问题，优先级仍应保持 P1。

关键上下文：
- 当前状态：in_progress
- 当前责任人：frontend_owner
- 最近关键评论 / 流转：最近两次评论都在讨论图片预览弹窗和上传过程遮罩，说明核心问题已收敛到交互层。

建议下一步：
- 先确认弹窗宽度、鼠标手势和过渡动画是否全部符合预期。
- 如果这三个点都已稳定，再流转到待确认或指派给测试同学复验。
```

## 3. 执行结果样例：已发表评论

适用场景：

- Codex CLI 已调用 `yuance_create_work_item_comment`

示例：

```text
已执行动作：
- 已在 YCE-BUG-18 下追加评论，说明图片预览弹窗的修复点和建议回归项。

目标对象：
- item_key：YCE-BUG-18

关键参数：
- body_format：html
- parent_comment_id：空

执行结果：
- 评论已创建成功，当前讨论串里已补充本轮修复说明。

建议下一步：
- 如果你希望同时推进流程，可以继续指定目标状态或处理人，我再执行流转 / 指派。
```

## 4. 执行结果样例：已流转并指派

适用场景：

- Codex CLI 已调用 `yuance_handoff_work_item`

示例：

```text
已执行动作：
- 已将 YCE-TASK-9 流转到 pending_confirmation，并指派给 qa_owner。

目标对象：
- item_key：YCE-TASK-9

关键参数：
- status：pending_confirmation
- assignee_username：qa_owner

执行结果：
- 工作项已成功流转，当前责任人已更新为 qa_owner。

建议下一步：
- 建议让测试同学重点确认消息角标、帖子跳转定位和已读状态是否一致。
```

## 5. 阻塞说明样例：缺少资料密码

适用场景：

- 资料 `is_protected = true`
- 用户还没有授权和提供密码

示例：

```text
当前阻塞点：
- 这条项目资料受访问密码保护，当前不能直接读取正文。

已确认信息：
- project_key：YCE
- resource_id：27
- 资料标题：正式环境接口配置

缺失信息：
- 该条资料的访问密码
- 用户对本次解锁的明确授权

建议补充：
- 如果你确认要查看这条资料，请直接提供这条资料的访问密码，我再调用解锁工具读取正文。
```

## 6. 阻塞说明样例：缺少目标状态

适用场景：

- 用户要求“处理一下这个工作项”
- 但没有明确希望评论、流转、关闭还是指派

示例：

```text
当前阻塞点：
- 当前可以继续执行写入，但缺少明确的目标动作。

已确认信息：
- item_key：YCE-BUG-22
- 当前状态：in_progress
- 当前责任人：frontend_owner
- 评论上下文已读取完成

缺失信息：
- 你希望我执行的是“发表评论”、“流转状态”还是“指派给他人”
- 如果要流转，目标状态是什么
- 如果要指派，目标处理人是谁

建议补充：
- 请直接告诉我你的目标动作，例如：
  - “回复评论，不改状态”
  - “流转到 pending_confirmation”
  - “指派给 qa_owner 并流转到 in_progress”
```

## 7. 通知梳理样例

适用场景：

- Codex CLI 已读取通知列表
- 用户想知道当前优先处理什么

示例：

```text
结论：
- 当前最值得优先处理的是两条未读指派和一条被回复通知，其中 YCE-BUG-18 的优先级最高。

关键证据：
- YCE-BUG-18 同时出现在未读通知和最近讨论里，说明它既有新上下文又仍在处理中。
- YCE-TASK-9 已经进入待确认，优先级低于仍在开发中的缺陷项。

当前风险 / 缺口：
- 如果先忽略 YCE-BUG-18，图片预览相关回归可能继续阻塞测试确认。

建议下一步：
- 先打开 YCE-BUG-18 继续看最新评论。
- 再处理 YCE-TASK-9 的确认或回退。
```

## 8. 使用建议

如果后续你发现 Codex CLI 输出风格不稳定，优先检查三份文档是否一致：

```text
skills/yuance-agent/SKILL.md
docs/mcp/ai-agent-playbook.md
docs/mcp/agent-output-examples.md
```

推荐关系：

- `SKILL.md`：定义规则
- `ai-agent-playbook.md`：解释规则
- `agent-output-examples.md`：展示规则的实际输出样子
